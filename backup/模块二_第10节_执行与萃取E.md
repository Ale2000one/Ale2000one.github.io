# 第10节：执行与萃取(E)——做，然后把经验变成知识

> 模块二：RIDE循环拆解与实战

---

## 本节目标

本节课程结束后，你将能够：

1. **理解**为什么"做"是不可跳过的学习环节
2. **掌握**执行阶段的四个核心动作
3. **应用**萃取的三个维度提炼经验
4. **产出**可复用的萃取成果（知识库条目、复盘文档等）
5. **完成**一次实践任务的完整闭环

## 开场：为什么RIDE的"E"是关键

在前面几节中，我们完成了RIDE循环的三个步骤：

- **R（Research）**：侦察，快速看清陌生领域的全貌
- **I（Identify）**：定位，找到"不解决就不动"的卡点
- **D（Deep-dive）**：深钻，窄而深地攻克一个知识点

现在，你已经：
- 对领域有了初步认知
- 定位到了关键卡点
- 深入学习了核心知识

**下一步是什么？**

很多人到这一步就停了——觉得"我学了，我懂了"，然后转向下一个目标。

但这恰恰是学习效果打折扣的关键时刻。

因为**"学"和"用"之间，隔着一道巨大的鸿沟**。

你可能：
- 学会了SQL的JOIN怎么写，但不知道什么时候该用
- 理解了留存分析的概念，但不知道如何应用到工作中
- 学了很多沟通技巧，但实际沟通时还是老样子

这道鸿沟，只能通过**"做"来跨越**。

**执行与萃取（Execute & Extract）**是RIDE循环的最后一步，也是最关键的一步——它让你的学习从"知道"变成"能用"，从"能用"变成"可复用"。

## 第一部分：为什么"做"是不可跳过的环节

### 1.1 知识必须在实践中被激活

认知科学研究表明，知识分为两种状态：

- **陈述性知识**：知道"是什么"，存储在脑子里，可以复述
- **程序性知识**：知道"怎么做"，内化为技能，可以自动化执行

传统的学习方式大量积累的是陈述性知识——你"知道"很多东西，但这些知识只是"存储"在你脑中，没有被"激活"。

只有通过实践，这些知识才会被激活，转化为程序性知识。

**举例**：

你看了10篇文章学习"如何做用户访谈"，这是陈述性知识——你知道有这些方法。

但当你真正去做一次用户访谈时，你会发现：
- 理论中的"访谈技巧"在真实场景中完全不同
- 遇到了很多文章里没提到的问题
- 原本以为"懂了"的东西，其实还有很多模糊的地方

这就是实践激活知识的过程。

### 1.2 做中学是最深刻的学

"做中学"（Learning by Doing）是最有效的学习方式之一。

**为什么做中学更有效？**

1. **问题驱动**：在做中学，你是带着具体问题去学的，学习的针对性和动力都更强。

2. **即时反馈**：在做中学，你马上就能知道学得对不对、效果好不好，不需要等待外部评估。

3. **情境编码**：在做中学，知识会被编码到具体的场景中，记忆更深刻，提取更容易。

4. **能力内化**：在做中学，知识会逐渐内化为技能，从"知道怎么做"变成"自动化地做"。

**传统学习 vs 做中学**：

| 维度 | 传统学习 | 做中学 |
|------|----------|--------|
| 学习顺序 | 先学后用 | 在用中学 |
| 知识状态 | 存储状态 | 激活状态 |
| 学习动力 | 外部驱动 | 问题驱动 |
| 效果转化 | 低 | 高 |
| 遗忘速度 | 快 | 慢 |

### 1.3 执行中发现的新问题是最好的学习素材

在执行过程中，你会发现很多"之前没想到"的问题。

这些问题是最有价值的学习素材，因为它们：
- 来自真实场景，迫切需要解决
- 暴露了你知识中的盲区
- 为下一轮学习指明了方向

**举例**：

你在学SQL时，理论上理解了JOIN操作。

但在实际提取数据时，你发现：
- 不知道数据存在哪张表里（数据发现能力不足）
- 不知道字段的命名规则（业务理解不够）
- 不知道NULL值的处理方式（细节知识缺失）

这些问题，会成为你下一轮"定位→深钻"的核心内容。

**执行是学习的开始，不是学习的终点。**

## 第二部分：执行阶段的四个动作

### 2.1 动作一：启动——克服"不知道从何下手"

**问题**：

学了很多知识，但真正要开始做的时候，反而不知道从哪里下手。

这是执行阶段的第一道坎。

**为什么会这样？**

- 知识太抽象，不知道怎么转化为具体行动
- 担心做不好，想"再准备准备"
- 完美主义作祟，想"一步到位"

**如何启动？**

**方法一：从最小行动开始**

不要想着"一步到位"，先找一个最小、最简单的行动开始做。

- 不是"完成一个完整的报告"，而是"先把数据提取出来"
- 不是"做一次完美的用户访谈"，而是"先列5个问题"
- 不是"搭建一个完整的数据分析体系"，而是"先画一个最简单的图"

**方法二：设定"启动仪式"**

给自己一个简单的启动仪式，打破拖延心理：
- "我只需要做15分钟"——降低心理门槛
- "我只需要做一点点"——减少压力
- "做砸了也没关系"——消除完美主义恐惧

**方法三：借用已有框架**

很多领域有现成的执行框架，可以借用：
- 项目管理中的"先做再迭代"
- 写作中的"先写再改"
- 设计中的"先画草图再优化"

**关键心态**：

> **启动的关键不是"准备好"，而是"先开始"。**
>
> 在做的过程中发现问题、解决问题，比等准备好了再做更有效。

### 2.2 动作二：推进——在错误中迭代

**问题**：

启动之后，过程中会遇到各种困难和错误。

很多人遇到错误就停下来，或者干脆放弃。

**为什么会这样？**

- 把错误视为"失败"，而不是"学习机会"
- 追求一步到位的完美
- 缺乏迭代的心态

**如何推进？**

**方法一：把错误当作信息**

错误不是失败，而是信息——它告诉你：
- 哪里有问题
- 什么方法不奏效
- 下次可以怎么改进

遇到错误时，问自己：
- "这个错误告诉我什么？"
- "我可以从中学到什么？"
- "下次如何避免同样的错误？"

**方法二：设置"迭代节点"**

不要想着一次做到完美，而是分阶段迭代：
- 1.0版本：做到60分，有基本功能就行
- 2.0版本：根据反馈改进
- 3.0版本：继续优化完善

每个版本都是一个迭代节点，帮你把大目标分解为可执行的小步骤。

**方法三：用"实验心态"做事情**

把自己正在做的事情当作一个实验：
- 假设：这个方法可行
- 验证：实际做出来看看
- 结果：成功还是失败？
- 迭代：调整假设，再试一次

**关键心态**：

> **错误不是终点，而是学习路上的里程碑。**
>
> 每犯一个错误，你就离正确答案更近一步。

### 2.3 动作三：验证——确认产出达到要求

**问题**：

做完之后，如何判断"做得怎么样"？

很多人做完就结束了，没有验证环节，不知道产出是否达标。

**为什么要验证？**

- 确保解决了原本的问题
- 发现还存在的不足
- 为下一轮迭代提供方向

**验证的三个维度**：

**维度一：问题解决度**

原定要解决的问题，现在解决了吗？
- 如果是，解决到什么程度？
- 如果不是，还差什么？

**维度二：质量达标度**

产出的质量达到要求了吗？
- 如果是，达到了什么标准？
- 如果不是，差在哪里？

**维度三：学习目标达成度**

通过这次执行，我学到了什么？
- 哪些知识被激活了？
- 哪些技能被锻炼了？
- 还有哪些不足需要继续学习？

**验证的方法**：

1. **自我检验**：对照标准检查自己的产出
2. **他人反馈**：让别人评价你的产出
3. **实际应用**：把产出放到实际场景中检验
4. **对比复盘**：和之前的状态对比，看有没有进步

### 2.4 动作四：复盘——总结经验教训

**问题**：

做完之后，如果没有系统性地总结，经验就会流失。

下次遇到类似问题，又要从头开始。

**为什么要复盘？**

复盘能帮你：
- 把具体经验提炼为可复用的知识
- 发现过程中的问题和改进点
- 加深对知识的理解和记忆

**复盘的三个层次**：

**层次一：做了什么**

描述执行过程中发生了什么：
- 我做了什么？
- 结果如何？
- 遇到了什么问题？

**层次二：学到了什么**

从具体经历中提炼规律：
- 哪些做法有效？为什么？
- 哪些做法无效？为什么？
- 有什么新发现？

**层次三：下次怎么做**

基于学习，制定改进行动：
- 下次遇到类似情况，我应该怎么做？
- 有什么需要提前准备的？
- 有什么坑要避开？

**复盘的格式示例**：

```
【复盘】用户访谈项目

【做了什么】
- 做了3次用户访谈，每次1小时
- 准备了10个问题，实际用了8个
- 录音并整理了访谈记录

【结果如何】
- 获得了有价值的用户反馈
- 但第2次访谈时有些问题效果不好
- 整体收获超出预期

【学到了什么】
- 开放式问题比封闭式问题更能引发深入讨论
- 访谈前需要充分了解用户背景
- 追问比直接提问更能挖掘信息

【下次怎么做】
- 问题数量控制在6-8个
- 访谈前先做用户背景调研
- 准备一个"追问清单"备用
```

## 第三部分：萃取——把经验提炼成知识

### 3.1 什么是萃取

**萃取**（Extraction）是RIDE循环的最后一个环节，指的是从具体的实践经验中，提炼出具有普适性的知识。

为什么要萃取？

因为：
- 具体经验是"一次性"的，只能用一次
- 萃取后的知识是"可复用"的，能用多次
- 没有萃取，经验就只是经验，无法转化为能力

**举例**：

你做了一次用户访谈，收获了很多具体经验：
- 访谈了5个用户
- 问了10个问题
- 获得了很多反馈

如果不做萃取，这些经验就只是"我做过访谈"。

如果做了萃取，你会得到：
- 做用户访谈的最佳问题清单
- 访谈中常见的问题和应对方法
- 访谈记录整理的模板

这些萃取成果，可以在未来反复使用。

### 3.2 萃取的三个维度

萃取不是简单记录，而是从三个维度深入提炼：

**维度一：操作层——具体怎么做**

萃取在操作层面的做法：
- 具体步骤是什么？
- 有哪些关键动作？
- 有什么具体的技巧？

**示例**：

原始经验："做用户访谈时，要先建立信任感"

萃取后：
- "开场时先自我介绍，说明访谈目的和时间"
- "前5分钟聊些轻松话题，让用户放松"
- "不要直接问敏感问题，先从开放式问题开始"

**维度二：原理层——为什么有效**

萃取在原理层面的规律：
- 为什么这样做有效？
- 背后的逻辑是什么？
- 什么条件下有效？

**示例**：

原始经验："开场时先自我介绍效果很好"

萃取后：
- "建立信任感需要让对方知道'你是谁、你要做什么、多长时间'——这满足了人对确定性的需求"
- "当对方感到安全时，更愿意分享真实想法"

**维度三：边界层——什么时候失效**

萃取在边界层面的知识：
- 什么时候这样做不适用？
- 什么情况下会失效？
- 有什么注意事项？

**示例**：

原始经验："开放式问题比封闭式问题好"

萃取后：
- "开放式问题适合探索性访谈，但如果用户时间紧张，可能需要更高效的问题"
- "开放式问题可能导致访谈时间过长，需要把控节奏"
- "对于特别沉默的用户，封闭式问题可能更容易打开局面"

### 3.3 萃取的完整流程

```
第一步：收集原始材料
↓
访谈记录、项目文档、复盘笔记等

↓
第二步：识别有价值的经验
↓
这次经历中，什么是有价值的？
什么值得萃取？

↓
第三步：从三个维度提炼
↓
操作层：具体怎么做
原理层：为什么有效
边界层：什么时候失效

↓
第四步：格式化输出
↓
整理成可复用的形式

↓
第五步：存入知识库
↓
保存起来，方便以后查阅
```

## 第四部分：萃取成果的形式

### 4.1 个人知识库条目

把萃取成果整理成知识库条目，存入个人知识管理系统。

**知识库条目的结构**：

```
【知识库】用户访谈技巧

【核心要点】
- 开场建立信任
- 用开放式问题
- 追问挖掘信息

【操作步骤】
1. 准备访谈提纲（6-8个问题）
2. 开场自我介绍（2分钟）
3. 从轻松话题切入（3分钟）
4. 进入核心问题（30分钟）
5. 感谢并结束（2分钟）

【注意事项】
- 不要问"你觉得好不好"这类引导性问题
- 被访谈者沉默时，不要急着填补空隙
- 录音需要获得对方同意

【适用场景】
- 新产品用户调研
- 功能反馈收集
- 用户需求挖掘

【更新记录】
- 2024年1月：根据第3次访谈更新"追问技巧"
```

### 4.2 复盘文档

把萃取的成果整理成复盘文档，便于回顾和分享。

**复盘文档的结构**：

```
【项目复盘】XX数据分析项目

项目背景
- 目标：分析Q3季度活动效果
- 时间：2024年1月
- 参与：运营+数据分析师

执行过程
- 数据提取：用SQL从数据库提取活动数据和销售数据
- 数据处理：用Excel进行清洗和整理
- 数据分析：计算核心指标，分析活动效果

关键发现
- 活动带来的销售增量约15%
- 不同渠道的活动效果差异显著
- 周末活动的转化率高于工作日

经验教训
- 归因逻辑需要提前和数据团队对齐
- SQL基础很重要，能节省很多等待时间
- 活动效果分析需要关注同期对比

行动计划
- 下次活动前先确认归因模型
- 每周固定时间学习SQL
- 建立活动效果分析的模板
```

### 4.3 可复用的模板/检查清单

把萃取成果转化为可直接使用的模板或检查清单。

**检查清单示例**：

```
【用户访谈检查清单】

【访谈前】
□ 确定访谈目标和问题
□ 了解被访谈者背景
□ 准备访谈提纲（6-8个问题）
□ 预约时间和地点
□ 准备录音设备（获得同意）

【访谈中】
□ 开场自我介绍，说明目的和时间
□ 从轻松话题切入，建立信任
□ 用开放式问题，引导深入
□ 注意追问，挖掘信息
□ 控制节奏，不要超时
□ 做好记录（录音或笔记）

【访谈后】
□ 感谢被访谈者
□ 整理访谈记录（24小时内）
□ 分析访谈内容
□ 提炼关键发现
□ 更新知识库

【常见问题应对】
□ 被访谈者沉默 → 不要急着填补空隙，等一等
□ 问题跑偏 → 温和拉回主题
□ 时间不够 → 优先问核心问题
□ 信息太多 → 记录关键词，后续跟进
```

### 4.4 教学素材

把萃取成果整理成可以教给别人的素材。

**教学素材的价值**：

- 整理的过程本身就是深度学习
- 教学是检验理解的最佳方式
- 分享能获得反馈，帮助完善知识

**教学素材的形式**：

- 内部分享PPT
- 经验分享文章
- 操作指南文档
- 培训课件

## 第五部分：案例演示——完成一次用户访谈后的萃取

### 5.1 案例背景

小李是一名产品经理，公司让他负责新功能的用户调研工作。

他做了3次用户访谈，这是他第一次做用户访谈。

访谈结束后，他需要进行执行与萃取。

### 5.2 执行阶段回顾

**启动**：

一开始，小李不知道从何下手——访谈要怎么进行？问题要怎么问？

他先从最小行动开始：先列5个核心问题，其他的边做边加。

**推进**：

在访谈过程中，他遇到了很多问题：
- 第1次访谈时问的问题太封闭，用户回答很简短
- 第2次访谈时时间没控制好，超时了
- 第3次访谈时学会了追问，效果好很多

他把这些当作学习机会，没有放弃。

**验证**：

3次访谈结束后，他验证了效果：
- 收集到了有价值的信息
- 产出了用户需求文档
- 为产品决策提供了支持

**复盘**：

他进行了完整的复盘，总结了经验教训。

### 5.3 萃取过程

**第一步：收集原始材料**

- 3次访谈的录音和笔记
- 整理后的用户反馈文档
- 自己写的复盘记录

**第二步：识别有价值的经验**

- 开场建立信任很重要
- 开放式问题比封闭式问题好
- 追问能挖掘更多信息
- 时间控制是难点

**第三步：从三个维度提炼**

**操作层**：
- 开场要自我介绍，说明目的和时间（2分钟）
- 从轻松话题切入，再进入核心问题
- 问题要开放式，不要封闭式
- 追问技巧："您能举个例子吗？""为什么？"

**原理层**：
- 信任感让用户更愿意分享真实想法
- 开放式问题让用户有更多发挥空间
- 追问能帮助挖掘用户的真实需求

**边界层**：
- 开放式问题不适合时间紧张的情况
- 追问可能让访谈时间变长
- 用户沉默时不要急着填补空隙

**第四步：格式化输出**

整理成知识库条目和检查清单。

**第五步：存入知识库**

保存到个人知识管理系统中，方便以后查阅。

### 5.4 萃取成果

**知识库条目：用户访谈技巧**

```
【知识库】用户访谈技巧

【核心要点】
1. 开场建立信任
2. 用开放式问题
3. 注意追问技巧
4. 控制访谈节奏

【操作步骤】
1. 准备：列6-8个问题，了解用户背景
2. 开场：自我介绍，说明目的（2分钟）
3. 切入：从轻松话题开始（3分钟）
4. 核心：问核心问题，注意追问（30分钟）
5. 收尾：感谢，总结要点（2分钟）

【注意事项】
- 不要问"你觉得好不好"这类引导性问题
- 用户沉默时等待，不要急着填补
- 开放式问题适合探索，追问挖掘信息
- 时间紧张时优先核心问题

【更新记录】

```

## 第六部分：本节总结

### 核心要点回顾

1. **为什么"做"不可跳过**：
   - 知识必须在实践中被激活
   - 做中学是最深刻的学
   - 执行中发现的问题是最好的学习素材

2. **执行阶段的四个动作**：
   - 启动：克服"不知道从何下手"——从最小行动开始
   - 推进：在错误中迭代——把错误当作学习机会
   - 验证：确认产出达到要求——检验问题解决度、质量、学习目标
   - 复盘：总结经验教训——做了什么、学到什么、下次怎么做

3. **萃取的三维度**：
   - 操作层：具体怎么做
   - 原理层：为什么有效
   - 边界层：什么时候失效

4. **萃取成果的形式**：
   - 个人知识库条目
   - 复盘文档
   - 可复用的模板/检查清单
   - 教学素材

### 关键认知

> **执行与萃取，是把学习成果转化为可复用能力的核心环节。**
>
> 没有执行，知识永远是"知道"；没有萃取，经验永远是"一次性的"。
>
> 做中学，边做边萃取，让每一次实践都成为能力升级的阶梯。
